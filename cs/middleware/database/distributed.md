# 分布式存储与分库分表

随着系统业务体量和数据量的快速增长，单机关系型数据库在 CPU、I/O 以及存储容量上面临物理瓶颈。为此，我们需要引入分布式存储架构与分库分表解决方案。

---

## 1. 拆分策略：垂直拆分 vs 水平拆分

分库分表主要包括**垂直拆分**与**水平拆分**两个维度。

| 拆分类型 | 维度 | 拆分方式 | 解决痛点 | 带来问题 |
| :--- | :--- | :--- | :--- | :--- |
| **垂直拆分 (Vertical)** | 按业务/列 | 将不同的业务表放入不同数据库（垂直分库），或将大表中的常用列和宽列（如 CLOB、TEXT）拆为多表（垂直分表）。 | 解决单数据库 I/O 争抢，优化内存利用率。 | 无法解决单表数据量过大带来的查询性能急剧下降。 |
| **水平拆分 (Horizontal)** | 按行数据 | 按照一定的分片算法（如取模或范围），将一张表的数据拆分到多个库的多张结构相同的表中。 | 解决单表数据量超限、单库连接数满和存储瓶颈。 | 路由复杂、跨库 Join 困难、分布式事务、分布式 ID 问题。 |

---

## 2. 分片键（Sharding Key）设计

水平拆分最核心的决策是**分片键（Sharding Key）**的选择。
- **好的分片键**应该能够将读写负载及存储容量**均匀地分摊**到各个物理节点。
- **常见分片算法**：
  1.  **哈希取模 (Hash Modulo)**：如 `user_id % 16`。数据分布极均匀，但扩容时需要重新迁移大量数据（可引入**一致性哈希**或**成倍扩容法**缓解）。
  2.  **范围划分 (Range-based)**：如按时间（每月一张表）或按主键范围（每 1000 万条数据一个分片）。利于扩容，但可能造成热点（如最新的时间分片写入压力极大）。

### 2.1 应对多维度查询的方案（如买家/卖家视角）
如果以 `buyer_id` 作为分片键，买家查询订单非常快。但如果卖家想要查询自己的所有销售订单，由于数据按 `buyer_id` 散落在各个库，需要进行**全表扫描/广播查询**，这会带来严重的性能问题。

**常用解决思路**：
- **数据异构法**：在卖家维度同步复制一份数据（利用 Canal 监听 Binlog 写入 Elasticsearch，或者利用独立的分片键存一份异构库）。
- **基因插入法**：在生成订单 ID 时，将 `seller_id` 的后几位“基因”融入订单 ID 的低位中，使得同一个订单 ID 即便包含买家 ID 也能通过计算路由到卖家所在的库。

---

## 3. 分布式唯一 ID 方案

在分库分表后，各个子表由于独立写入，不能再依赖单表自增主键（会造成多表主键冲突）。必须使用分布式唯一 ID 生成器。

### 3.1 常见 ID 生成方案比较
1.  **UUID**：
    - *优点*：本地生成，性能极高，无网络开销。
    - *缺点*：无序、过长（36 位字符）、占用空间大。作为 MySQL 主键会导致 B+ 树页分裂频繁，极大降低写入性能。
2.  **数据库号段模式 (Segment)**：
    - *优点*：自增，方便步长控制。
    - *缺点*：依然有数据库单点请求的潜在压力（通常引入 proxy 批量拉取号段缓存在本地）。
3.  **Snowflake (雪花算法)**：
    - Twitter 开源的分布式 ID 生成算法。它生成的是一个 **64 位的 Long 型数字**。

#### 雪花算法结构分析：
```
 1bit     41bit (毫秒时间戳)        10bit (机器ID)   12bit (序列号)
┌───┐┌─────────────────────────┐┌──────────────┐┌────────────┐
│ 0 ││ 00000000000000000000... ││ 0000000000   ││ 0000000000 │
└───┘└─────────────────────────┘└──────────────┘└────────────┘
```
- **1 位符号位**：固定为 0，保证生成的 ID 为正数。
- **41 位时间戳**：精确到毫秒，可支持约 69 年。
- **10 位工作机器 ID**：支持最多 1024 个节点。
- **12 位序列号**：同一个毫秒内生成的计数累加，每毫秒每节点支持最多 4096 个唯一 ID。

*雪花算法的核心缺陷*：依赖系统时钟。如果系统**时钟回拨**，可能生成重复的 ID（对此一般需要检测时钟回拨并抛出异常，或者等待时钟追回后再生成）。
